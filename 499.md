## ABSTRACT  

BlockMed Version Three Blockchain Based Prescription and Medicine Verification Platform  

Fake medicine is a danger for public health in many places and Bangladesh also faces illegal medicine circulation for long time. We often see discussion in newspapers and television. Weak prescription proof can cause argument between doctor, patient and pharmacy. So we planned one system where prescription and medicine batch information can remain in a ledger that cannot be secretly edited easily. Blockchain type idea matches this type of expectation because after confirmation changing history silently becomes difficult compared with normal centralized database misuse story.  

We built a web software where doctors can create prescriptions, pharmacies can verify and dispense, manufacturers can relate batch information with chain record, and admins can supervise users depending on requirement. Sensitive patient naming style data is avoided on chain mainly we keep hash reference and details can stay off chain portion as structured data. Patients can trace own related view in simplified mode in our demonstration. Role wise access is arranged so unrelated user cannot misuse another role action in normal flow.  

We followed normal software engineering steps: requirement clarification from coursework scope, modular web development using React toolchain, Ethereum style smart contracts using Solidity compiled and deployed via Hardhat, and linking browser wallet signing or development mode signer for demonstration. Implementation is tested mainly by manual scenario testing checklist and iterative debugging on localhost network before writing final results.  

We obtained working demonstration where issuing prescription prints transaction identification, barcode style quick response picture generates so pharmacy section can lookup, dispensing path executes when pharmacist role behaves according to workflow, medicine batch lifecycle functions activate as contract permits including recall and suspicion flag pathway in present repository version. Finally output shows decentralized prescription tracing idea workable at academic demonstration level without claiming commercial hospital rollout yet. Practical meaning is undergraduate students gain clearer picture how accountability in healthcare workflow can relate with decentralized record thinking when future policy guideline and ethics review will be matured for real-world deployment scenario.

## Chapter 4 Investigation/Experiment, Result, Analysis and Discussion

We tested BlockMed on laptops in the laboratory. We did not experiment on real hospital patients because our senior design remained software demonstration only and we had no ethical clearance for patient studies.

A practical issue we faced repeatedly was forgetting to paste the newest contract address into the frontend dot environment file after redeploying Solidity. The UI sometimes looked normal while transactions failed or seemed random until the deployment address matched Hardhat output again. Documentation in the repository about metamask localhost network mismatch also saved our time.

We followed a simple repeated routine like a manual checklist (not advanced measurement lab style work).

Restart local ethereum test-chain service.

Rebuild and redeploy Solidity contract whenever contract code changes.

Update frontend environment configuration with the Hardhat printed contract address after every redeploy.

Register demo wallets for Doctor, Pharmacist and Manufacturer after chain reset wipes old accounts.

Doctor issues prescription using dummy hashed patient token.

Pharmacy verifies by typing prescription identifier or scanning barcode quick response image when camera permission allows.

Complete dispensing and inspect event style feedback; optionally try manufacturer suspicion-flag and recall subplot exercises for extra confidence.

Expected success looks like visible transaction hexadecimal identifier with confirmation toast notification. MetaMask wrong-network rejections happened sometimes during testing; however internal troubleshooting notes reduced confusion noticeably.

From this experience we understand that cryptographic design cannot fix sloppy deployment habits misleading an examiner. Offline or demo mode is useful for classroom evaluation but it is not the same as full decentralisation story, so limitation is discussed again in Chapter Eight. We did not publish formal latency or gas spreadsheet benchmarks; that can remain future extension work.

## Chapter 5 Impacts of the Project

### 5.1 Impact of this project on societal, health, safety, legal and cultural issues

Counterfeit medicine hurts public health. Bangladesh media frequently reports medicine adulteration and illegal supply stories, so a student project that talks about traceability can add modest awareness value even though nationwide adoption is far beyond our scope.

If pharmacists truly follow verification steps before dispensing, patient safety can improve in theory; still human carelessness remains a real loophole and we mention that honestly.

Healthcare data rules in Bangladesh are evolving and institutions must obey proper legal advice. Blockchain history is hard to delete compared with ordinary database delete operations, so naive privacy expectations need careful explanation. Many senior pharmacy staff members may feel nervous about metamask style wallet flows, so cultural training and empathy are important if such systems are introduced in real life rather than mockery from young engineers.

### 5.2 Impact of this project on environment and sustainability

Electronic workflow can reduce paper stamping if organisations digitise sincerely in future, but large-scale adoption is uncertain for our prototype level work.

There is public debate about electricity use of some public blockchain mining models. For our coursework we mostly used local ephemeral Hardhat rehearsal node, so hardware electricity footprint was small compared with heavy production mining narrative, and we keep that contrast in mind when talking about sustainability honestly.
